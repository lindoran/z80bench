# z80bench — Design Document

Z80 ROM annotation workbench. A C library backend with a native desktop GTK4 UI.
The library owns all project logic; the UI links against it directly.
No HTTP server, no browser, no serialisation between backend and UI.

The immediate goal is a good enough tool to reverse engineer the VZ200 ROM.
Scope is intentionally tight — what z80dasm and z88dk already provide, plus a
clean annotation layer on top. Nothing is reinvented that a tool already handles.

---

## Target Stack

| Layer | Choice |
|-------|--------|
| Disassembler | z80dasm (subprocess) |
| Assembler output | z88dk / z80asm syntax |
| Backend | C library (`libz80bench`) |
| UI | GTK4 — dark mode, cross-platform, Linux/Plasma/Windows |
| Build | Make, mingw cross for Windows |

GTK4 is the right call: native on Linux (works under both Plasma and GNOME),
available on Windows via MSYS2, good dark mode support, and the widget set maps
naturally to a scrolling workbench layout. Qt is an alternative if GTK4 proves
problematic on any target.

The UI layer is a thin shell. All logic lives in the library. A different UI
(Qt, terminal, or a VS Code extension via a small stdio shim) can be dropped
on top without touching the library.

---

## Concepts

**ROM** — the binary under study. Loaded once, never mutated. Stored verbatim as
`rom.bin`.

**Region** — a contiguous address range with an interpretation: `CODE`, `DATA_BYTE`,
`DATA_WORD`, `DATA_STRING`, or `GAP`. The disassembler only operates on CODE regions.
Regions must not overlap.

**Annotation** — a per-address record in `annotations.ann`: optional label, optional
inline comment, optional block comment, optional xref note.

**Segment Entry** — a named address range in `segments.map` describing the
system's logical layout (ROM, RAM, VRAM, I/O, etc.). Used for display context
and region control. The old "memory map" name survives in the file name, but
the UI and docs now treat these as segments.

**Symbol** — a named address or constant in `symbols.sym`, tagged with a type
that drives both assembler EQU output and UI colour highlighting.

**Project** — a directory containing five flat files. Simple, diffable,
hand-editable in a pinch.

## Project File Layout

```
myproject/
    rom.bin         # ROM binary — read-only after import
    listing.mnm     # disassembly: mnemonics keyed by byte offset — derived, regenerable
    annotations.ann # annotations: labels, comments, block notes, region types
    segments.map    # segment map: named address ranges with types
    symbols.sym     # symbols: named addresses/constants with type tags
```

Legacy `project.*` filenames are still accepted on load for existing projects,
but new projects and saves use the clearer names above.

All text files are UTF-8, line-oriented, `#` or `;` for full-line comments
(`;` in `.sym` for z88dk compatibility, `#` elsewhere).
All addresses are hex with `0x` prefix.
Byte offsets in `.mnm` and `.ann` are decimal integers — direct index into the
ROM blob, independent of load address.

## File Formats

### rom.bin

Raw binary. No header. Loaded at the address specified by `load_addr` in the
`[meta]` block of `annotations.ann`. Never written by z80bench after initial import.

---

### listing.mnm

One record per disassembled unit (instruction or data pseudo-op).
Generated by `disasm.c`, regenerated on demand — **do not hand-edit**.
Exists so the UI can display the listing without invoking z80dasm on every
scroll or resize.

```
# z80bench listing
# Regenerate: z80bench --regen <projectdir>
# format: offset<TAB>byte_count<TAB>mnemonic<TAB>operands

0       3       JP      0x0050
3       1       DEFB    0x00
4       1       DEFB    0x00
80      3       CALL    0x1000
83      1       RET
```

Fields are tab-separated. `operands` is empty for instructions like `RET`, `NOP`.
`offset` is decimal — byte offset into `rom.bin`, not the load address.

The raw bytes displayed in the listing view are read directly from `rom.bin`
using `offset` and `byte_count`. They are not stored in `listing.mnm`.

---

### annotations.ann

Annotations keyed by byte offset. Also holds project metadata.
This is the primary file the user edits (via the UI, or directly if needed).

```
# z80bench annotation file

[meta]
name        = VZ200 ROM
load_addr   = 0x0000
version     = 1
line_limit  = 100

[region]
offset      = 0
end         = 16383
type        = CODE

[region]
offset      = 256
end         = 511
type        = DATA_BYTE

[label]
offset      = 0
name        = RESET

[label]
offset      = 80
name        = INIT

[comment]
offset      = 0
text        = Jump to main initialisation routine

[block]
offset      = 80
text        = Hardware initialisation. Called from RESET and WARMSTART.
              Destroys AF, BC, DE, HL.

[xref]
offset      = 80
text        = RESET, WARMSTART
```

**Section types:**

| Section | Key fields | Notes |
|---------|-----------|-------|
| `[meta]` | name, load_addr, version, line_limit | One block per project |
| `[region]` | offset, end, type | type: CODE, DATA_BYTE, DATA_WORD, DATA_STRING, GAP |
| `[label]` | offset, name | Assembler label for this address |
| `[comment]` | offset, text | Inline comment on the instruction |
| `[block]` | offset, text | Multi-line block comment printed above the label |
| `[xref]` | offset, text | Free-text cross-reference note |

Long `text` values wrap with leading whitespace on continuation lines (INI-style).

---

### segments.map

Segment map. Named address ranges with a type. Used for display context in the
UI and by the segment-to-region reconciler to control code/data layout.

Direct-byte and define-message segments may sit inside broader CODE or DATA
ranges. They do not overlap one another. When a segment is added, the broader
regions are split around it.

```
# z80bench segment map
# format: name<SPACE>start<SPACE>end<SPACE>type<SPACE>notes
# types: ROM  RAM  VRAM  IO  SYSVARS  UNMAPPED

ROM         0x0000  0x3FFF  ROM       VZ200 ROM
RAM         0x7800  0x7FFF  RAM       System RAM
IO          0x00FE  0x00FE  IO        Keyboard port
```

Fields are whitespace-separated. `notes` is everything after the fourth field.
Ranges may overlap where the hardware warrants it.

---

### symbols.sym

Symbol table. Directly includable by z88dk (`INCLUDE "symbols.sym"`).
Type tags appear in comment headers that z88dk ignores but z80bench parses
for colour highlighting.

```
; z80bench symbol file
; assembler: z88dk/z80asm
; INCLUDE this file at the top of your exported .asm

; [ROM_CALL]
CLS             EQU     0x0D6B      ; Clear screen
PRINT_AT        EQU     0x0D78      ; Print at row/col

; [VECTOR]
RST_08          EQU     0x0008
RST_10          EQU     0x0010
RST_18          EQU     0x0018
RST_20          EQU     0x0020
RST_28          EQU     0x0028
RST_30          EQU     0x0030
RST_38          EQU     0x0038

; [JUMP_LABEL]
INIT            EQU     0x0050
MAIN_LOOP       EQU     0x0100

; [WRITABLE]
SCORE           EQU     0xC000
LIVES           EQU     0xC002

; [PORT]
KBD_PORT        EQU     0xFE

; [CONSTANT]
SCREEN_WIDTH    EQU     32
SCREEN_HEIGHT   EQU     24
```

**Symbol types and UI highlight colours:**

| Tag | Meaning | Colour |
|-----|---------|--------|
| `ROM_CALL` | Calls into ROM routines | Orange |
| `VECTOR` | RST and interrupt vectors | Yellow |
| `JUMP_LABEL` | Internal jump/call targets | Green |
| `WRITABLE` | RAM addresses | Cyan |
| `PORT` | I/O port numbers | Magenta |
| `CONSTANT` | Pure numeric constants | White / default |

## Library API Overview

The backend is a C library (`libz80bench`). The GTK4 UI calls into it directly.
Full signatures are in `z80bench.h`. This section describes module responsibilities.

| Module | File | Responsibility |
|--------|------|----------------|
| Project | `project.c` | Open/create project dir, coordinate load/save of all five files |
| Disassembly | `disasm.c` | Invoke z80dasm, parse output into DisasmLine[], write/read `.mnm` |
| Annotations | `annotate.c` | Parse/write `.ann`, CRUD for all section types, `xref_build()` |
| Memory map | `memmap.c` | Parse/write `.map` |
| Symbols | `symbols.c` | Parse/write `.sym`, type lookup, colour tag resolution |
| Export | `export.c` | Produce annotated listing (`.lst`) and z88dk source (`.asm`) |

**`xref_build()`** — a single O(n) pass over the `DisasmLine[]` array that
collects two things into a `XrefEntry[]` array:

1. Every unique `operand_addr` (addresses referenced by instructions) with a
   reference count and the matching symbol name if one exists.
2. Every `RTYPE_ORPHAN` line — bytes z80dasm could not decode — recorded by
   their first-byte address with entry type `ORPHAN`.

Requires no file I/O, no subprocess, and no new file format. Called once after
every Regen and whenever the symbol table changes. The References panel in the
UI is built directly from this array.

The `XrefEntry` struct:

```c
typedef enum {
    XREF_ADDR,    /* address referenced by one or more instructions */
    XREF_ORPHAN   /* byte z80dasm could not decode */
} XrefType;

typedef struct {
    int      addr;                        /* absolute address */
    XrefType xtype;
    int      ref_count;                   /* number of instructions referencing this addr; 0 for ORPHAN */
    char     sym_name[DISASM_LABEL_MAX];  /* symbol name if known, else "" */
    int      sym_type;                    /* SymbolType enum value, -1 if none */
    int      in_rom;                      /* 1 if addr falls within the ROM range */
    unsigned char first_bytes[3];         /* first 1-3 bytes from rom.bin if in_rom */
    int      first_byte_count;
} XrefEntry;
```

No HTTP. No JSON. No runtime serialisation between UI and library.

## Export Formats

Both exports are triggered from the UI and written to a path the user chooses.

### Annotated Listing (`.lst`)

Human-readable. Fixed-width columns. For printing or review.

```
; ============================================================
; ROM  0x0000 - 0x3FFF  (VZ200 ROM)
; ============================================================

; ---- RESET ------------------------------------------------
; Jump to main initialisation routine
0000  C3 50 00    JP   INIT

; ---- INIT -------------------------------------------------
; Hardware initialisation. Called from RESET and WARMSTART.
; Destroys AF, BC, DE, HL.
; xref: RESET, WARMSTART
0050  CD 6B 0D    CALL CLS
0053  21 00 40    LD   HL,0x4000
```

Column layout (widths based on `line_limit` setting):
`addr(4)  bytes(8)  mnemonic(5)  operands(20)  ; comment`

Orphan lines append `; *** ORPHAN ***` as their comment. GAP lines append
`; gap`. Both are valid z88dk syntax in the `.asm` export.

### Assembler Source (`.asm`)

z88dk-compatible. Directly assemblable.

```asm
; z80bench export — VZ200 ROM
; Generated: 2026-03-20
; Assembler: z88dk/z80asm

        INCLUDE "symbols.sym"
        ORG     0x0000

; ============================================================
; ROM  0x0000 - 0x3FFF
; ============================================================

; Hardware initialisation. Called from RESET and WARMSTART.
; Destroys AF, BC, DE, HL.
; xref: RESET, WARMSTART
RESET:  JP      INIT
        DEFB    0x00

INIT:   CALL    CLS
        LD      HL,0x4000
```

```
z80bench/
    src/
        main.c          # GTK4 app entry point
        ui_listing.c    # scrolling listing view widget

        export.c        # .lst and .asm export

```

## Build Targets

```makefile
make              # native Linux, GTK4 from system packages
make windows      # mingw cross-compile, GTK4 bundle required
make install      # install binary + z80dasm dependency check
```

No vendored C dependencies. GTK4 from system packages.
z80dasm must be installed and in PATH (or placed alongside the binary).

## Design Constraints

- **ROM is immutable.** `rom.bin` is never written after import.
- **`.mnm` is derived.** Always regenerable from `rom.bin` + region
  definitions in `annotations.ann`. Users never edit it directly.
- **`.ann`, `.map`, `.sym` are the user's work.** Plain text, diffable,
  readable and editable in any text editor.
- **Bytes come from the ROM blob.** The `bytes[]` array in `DisasmLine` is
  populated by reading `rom.bin` directly, using the `offset` and
  `byte_count` fields. z80dasm is not the source of byte data.
- **Use what the tools provide.** z80dasm supplies the mnemonic and operand
  text. z88dk defines the output syntax. z80bench adds only the annotation
  layer — it does not reimplement anything either tool already does.
- **Dark mode by default.** GTK4 honours the system theme; the app requests
  a dark colour scheme on startup.
- **Line length limit** is configurable per project in `[meta]`. Default 100.
- **Assembler target is z88dk.** No abstraction for other assemblers in v1.
- **No runtime dependencies beyond GTK4 and z80dasm.**
- **Colour spelling.** All source comments, UI strings, and documentation use
  British spelling: colour, not color.

## Future / Out of Scope for v1

- Cycle count display — z80dasm does not output T-states. Would require a
  static opcode lookup table inside `disasm.c`. Straightforward to add later.
- Flow classification and loop head detection — useful but not needed to read a ROM.
- Additional CPU targets (6809, 6502) with target-specific disassembler
- Cross-reference graph (auto-detect all callers of a label)
- Control flow graph visualisation
- ROM patching / modification
- Multi-file / banked ROM support
- VS Code extension (small stdio shim over the library, straightforward addition)

### v2 — Live Assembly / IDE Mode

Once v1 is stable the infrastructure already supports live assembly with minimal
new work. z88dk is invoked as a subprocess (same pattern as z80dasm) and results
feed back into the in-memory listing model.

**Inline edit mode** — clicking an instruction row makes the operands field
editable. On each keystroke z88dk assembles the single line; the bytes column
updates live. Errors shown inline in red. Invalid assembly prevents commit.

**Assembly scratch pad** — a small freeform panel where the user types arbitrary
Z80 assembly and sees the resulting bytes alongside each line in real time. Useful
for drafting patch code and figuring out byte sequences to search for in the ROM.

Both features share one new module: `assemble.c` — a thin z88dk subprocess
adapter symmetric with `disasm.c`. Input is a line or block of z88dk-syntax
assembly; output is a byte array and error string.
